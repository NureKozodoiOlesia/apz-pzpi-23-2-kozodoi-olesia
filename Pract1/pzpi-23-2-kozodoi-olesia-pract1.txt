3 ОПИС ВИКОНАНОЇ РОБОТИ


3.1 Загальні відомості про шаблони (патерни) проєктування


Потреба у використанні шаблонів проєктування виникає через необхідність стандартизації підходів до вирішення типових архітектурних задач у розробці програмного забезпечення. Під час створення програмних систем без застосування загальноприйнятих рішень розробники нерідко витрачають зусилля на повторне розв'язання вже відомих проблем. Такий підхід ускладнює архітектуру, знижує якість коду та ускладнює його подальшу підтримку і масштабування.
Патерни проєктування — це перевірені часом концептуальні рішення для поширених архітектурних проблем. Вони не є готовим фрагментом коду, а являють собою вищий рівень абстракції, що визначає правила організації взаємодії класів та об'єктів. Крім того, спільне використання патернів формує єдину термінологічну базу серед розробників: назва конкретного шаблона одразу передає колегам закладений архітектурний підхід.
Класичну класифікацію шаблонів проєктування запропонували автори книги «Прийоми об'єктно-орієнтованого проєктування» (Gang of Four, GoF). Вони виокремили двадцять три основні патерни, розподілені за трьома категоріями:
* Породжувальні шаблони (Creational patterns) — зосереджені на механізмах створення об'єктів. Забезпечують незалежність системи від способу ініціалізації об'єктів та приховують складну логіку їх конструювання. До цієї групи належать: Абстрактна фабрика, Будівельник, Фабричний метод, Прототип, Одинак.
* Структурні шаблони (Structural patterns) — відповідають за організацію та компонування об'єктів і класів у більші гнучкі структури. Забезпечують ефективну підтримку зв'язків між елементами системи. До цієї групи належать: Адаптер, Міст, Компонувальник, Декоратор, Фасад, Легковаговик, Замісник.
* Поведінкові шаблони (Behavioral patterns) — визначають алгоритми взаємодії між об'єктами та способи розподілу відповідальності. До цієї групи належать: Ланцюжок відповідальності, Команда, Ітератор, Посередник, Знімок, Спостерігач, Стан, Стратегія, Шаблонний метод, Відвідувач, Інтерпретатор.
Патерни проєктування застосовуються на різних етапах розробки об'єктно-орієнтованого програмного забезпечення незалежно від мови програмування. Вони слугують фундаментом для побудови потужних фреймворків, стандартних бібліотек та корпоративних систем. Особливо корисними шаблони є там, де вимагаються висока гнучкість, розширюваність та надійна ізоляція складної логіки від клієнтського коду.


3.2 Опис обраного патерна


Назва патерна: «Команда» (Command).
Тип шаблона (патерна): поведінковий (Behavioral).
Основне призначення: патерн «Команда» перетворює запити або операції на окремі об'єкти-команди. Завдяки цьому запити можна передавати як аргументи методів, зберігати у колекціях, ставити в чергу та скасовувати. Ключовою особливістю патерна є повне відокремлення об'єкта, що ініціює запит (Invoker), від об'єкта, що його виконує (Receiver). Між ними знаходиться команда — самодостатній об'єкт, що інкапсулює всю необхідну інформацію для виконання та скасування дії.
Використання патерна є доцільним у таких випадках:
* необхідно параметризувати об'єкти виконуваними операціями — наприклад, елементи UI мають реагувати на дії, не знаючи, яка конкретно операція буде викликана;
* операції необхідно ставити в чергу, планувати або виконувати асинхронно;
* потрібна підтримка операцій скасування та повторного виконання (Undo/Redo);
* необхідно реалізувати транзакційну поведінку, за якої набір операцій має виконуватися або скасовуватися як єдине ціле;
* потрібне журналювання операцій для їх подальшого відтворення або аудиту.


Переваги:
* дотримання принципу єдиної відповідальності (Single Responsibility Principle): класи-ініціатори операцій відокремлені від класів-виконавців;
* дотримання принципу відкритості/закритості (Open/Closed Principle): нові команди додаються без зміни наявного коду;
* природна підтримка операцій скасування та повторного виконання через збереження команд у стеку;
* можливість відкладеного й асинхронного виконання операцій;
* можливість складати прості команди у макро-команди.


Недоліки:
* збільшення кількості класів: кожна окрема операція потребує власного класу команди;
* надмірна складність для простих сценаріїв, де достатньо прямого виклику методу.


3.3 Структура шаблона (патерна)


Базова структура патерна «Команда», представлена на діаграмі (див. рис. 3.1), складається з чотирьох ключових компонентів, кожен з яких виконує чітко визначену роль:
— ICommand (Інтерфейс команди) — абстрактний інтерфейс, який визначає загальний контракт для всіх команд у системі. Він декларує два обов'язкові методи: Execute(), призначений для виконання операції, та Undo(), призначений для її скасування. Завдяки цьому інтерфейсу викликач може однаково працювати з будь-якою командою, не знаючи деталей її реалізації.
— ConcreteCommand (Конкретна команда) — клас, що безпосередньо реалізує інтерфейс ICommand. Він зберігає посилання на об'єкт-отримувач (Receiver) та інкапсулює всю логіку виконання конкретної операції. Метод Execute() делегує фактичну роботу отримувачу, а метод Undo() реалізує зворотню дію.
— Invoker (Викликач) — клас, що ініціює виконання команд. Він зберігає посилання на об'єкт ICommand і викликає його методи у потрібний момент. Викликач повністю відокремлений від конкретних отримувачів і не знає, яку саме роботу виконує команда. Додатково він може зберігати історію виконаних команд для підтримки операції скасування.
— Receiver (Отримувач) — клас, що містить реальну бізнес-логіку. Саме він виконує фактичні дії, до яких звертається конкретна команда через метод action(). Отримувач повністю ізольований від командного рівня і не знає про існування інтерфейсу ICommand.
Взаємодія компонентів патерна базується на принципі інкапсуляції запиту як окремого об'єкта. Клієнтський код створює екземпляр ConcreteCommand і передає йому посилання на відповідний Receiver. Готова команда передається до Invoker. Коли настає потрібний момент, Invoker викликає метод Execute() команди. ConcreteCommand, своєю чергою, делегує виконання фактичної роботи отримувачу через відповідний метод. Усі виконані команди можуть накопичуватися в стеку. При необхідності скасування Invoker витягує останню команду зі стеку та викликає її метод Undo(), який відновлює попередній стан системи.
Наведена UML-діаграма класів (рис. 3.1) наочно відображає статичну архітектуру патерна. Пунктирна стрілка з порожнім трикутником позначає відношення реалізації — ConcreteCommand зобов'язаний виконати контракт інтерфейсу ICommand. Пунктирна стрілка зі звичайним наконечником від Invoker до ICommand позначає залежність — викликач взаємодіє лише з абстракцією, не знаючи конкретних реалізацій. Суцільна стрілка від ConcreteCommand до Receiver позначає асоціацію — конкретна команда зберігає посилання на свій отримувач і делегує йому виконання роботи.

3.4 Реалізація шаблона (патерна)


У рамках практичного завдання розроблено програму мовою C#, яка демонструє застосування патерна «Команда» на прикладі системи керування пристроями розумного будинку через пульт дистанційного керування. У реалізації роль ICommand виконує однойменний інтерфейс з методами Execute() та Undo(). Роль Receiver відіграють класи Light (освітлення з підтримкою назви кімнати) та Television (телевізор). Конкретні команди — LightOnCommand, LightOffCommand, TvOnCommand та TvOffCommand — відповідають ConcreteCommand і делегують виконання відповідним отримувачам. Роль Invoker виконує клас RemoteControl, який зберігає стек виконаних команд і забезпечує підтримку операції скасування Undo. Повний програмний код наведено у Додатку В. 


Основні етапи реалізації:


1.  Створення інтерфейсу команди.
Інтерфейс ICommand визначає єдиний контракт для всіх команд у системі. Він містить два методи: Execute() — для виконання дії, та Undo() — для її скасування. Завдяки такому інтерфейсу викликач може однаково звертатися до будь-якої команди без знання її деталей (рядки 1–5 Додатку В).


2. Реалізація класів-отримувачів.
Класи Light та Television є отримувачами — вони містять реальну логіку керування пристроями. Клас Light зберігає назву кімнати та надає методи TurnOn() і TurnOff(), що виводять відповідне повідомлення. Клас Television аналогічно реалізує методи увімкнення та вимкнення. Отримувачі повністю незалежні від командного рівня (рядки 7–24 Додатку В).


3. Реалізація конкретних команд.
Для кожної операції створено окремий клас, що реалізує ICommand. Клас LightOnCommand зберігає посилання на об'єкт Light і в методі Execute() викликає TurnOn(), а в Undo() — TurnOff(). Аналогічно побудовано LightOffCommand, TvOnCommand та TvOffCommand. Завдяки такій структурі додавання нової операції зводиться до створення нового класу команди без зміни наявного коду (рядки 26–56 Додатку В).


4. Реалізація Invoker — класу RemoteControl.
Клас RemoteControl є викликачем (Invoker). Він зберігає виконані команди у стеку типу Stack<ICommand>. Метод PressButton() виконує передану команду і додає її до стеку. Метод PressUndo() витягує останню команду зі стеку і викликає її Undo(), відновлюючи попередній стан системи. Якщо стек порожній — виводиться інформаційне повідомлення (рядки 58–80 Додатку В).


5. Клієнтський код.
У методі Main() класу Program створюються об'єкти пристроїв (отримувачів), потім — конкретні команди, до яких передаються посилання на відповідні пристрої. Команди передаються до пульта керування та послідовно виконуються. Демонструється як виконання команд, так і їх скасування за допомогою Undo. Зокрема, перевіряється ситуація, коли стек порожній і скасування вже неможливе (рядки 82–112 Додатку В).

  1  public interface ICommand
  2  {
  3      void Execute();
  4      void Undo();
  5  }

  7  public class Light

 10  
 11      public Light(string location)

 13          _location = location;
 14      }
 15  
 16      public void TurnOn()  => Console.WriteLine($"[Свiтло] {_location}: увiмкнено");
 17      public void TurnOff() => Console.WriteLine($"[Свiтло] {_location}: вимкнено");
 18  }

 20  public class Television
 21  {

 24  }

 26  public class LightOnCommand : ICommand
 27  {
 28      private readonly Light _light;
 29      public LightOnCommand(Light light) => _light = light;
 30      public void Execute() => _light.TurnOn();
 31      public void Undo()    => _light.TurnOff();
 32  }

 34  public class LightOffCommand : ICommand
 35  {
 36      private readonly Light _light;
 37      public LightOffCommand(Light light) => _light = light;
 38      public void Execute() => _light.TurnOff();
 39      public void Undo()    => _light.TurnOn();
 40  }

 50  public class TvOffCommand : ICommand
 51  {
 52      private readonly Television _tv;

 54      public void Execute() => _tv.TurnOff();
 55      public void Undo()    => _tv.TurnOn();
 56  }
 57  
 58  public class RemoteControl
 59  {
 60      private readonly Stack<ICommand> _history = new Stack<ICommand>();
 61  
 62      public void PressButton(ICommand command)
 63      {
 64          command.Execute();
 65          _history.Push(command);
 66      }
 67  
 68      public void PressUndo()
 69      {
 70          if (_history.Count == 0)
 71          {
 72              Console.WriteLine("[Пульт] Немає дiй для скасування.");
 73              return;
 74          }
 75  
 76          ICommand lastCommand = _history.Pop();
 77          lastCommand.Undo();
 78          Console.WriteLine("[Пульт] Дiю скасовано");
 79      }
 80  }
