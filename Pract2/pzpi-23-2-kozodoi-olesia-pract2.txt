3 ОПИС ВИКОНАНОЇ РОБОТИ


3.1 Загальні відомості про систему


Назва системи: Slack.
Розробник: Slack Technologies, Inc. (з 2021 року — підрозділ корпорації Salesforce, США).
Призначення системи: Slack — це хмарна платформа для командної комунікації та співпраці, орієнтована переважно на корпоративних користувачів. Система надає можливість обмінюватися повідомленнями в реальному часі, організовувати спілкування у тематичних каналах та приватних чатах, здійснювати голосові та відеодзвінки, обмінюватися файлами, а також інтегруватися з сотнями зовнішніх сервісів — GitHub, Jira, Google Drive тощо. Станом на 2025 рік платформа налічує понад 20 мільйонів щоденно активних користувачів і обробляє мільярди повідомлень на добу.
Slack виник із несподіваного джерела: команда розробників спочатку створювала браузерну MMORPG-гру під назвою Glitch. Хоча гра не мала комерційного успіху, внутрішній інструмент комунікації, розроблений для її команди, ліг в основу Slack. Перша версія сервісу запущена у 2013 році.


3.2 Архітектурний стиль системи


Slack використовує комбінований архітектурний підхід: мікросервісна архітектура у поєднанні з подієво-орієнтованою моделлю взаємодії між компонентами. На пізнішому етапі розвитку компанія перейшла також до стільникової (cellular) архітектури для підвищення відмовостійкості. Medium
Спочатку Slack розроблявся як класичний LAMP-стек (Linux, Apache, MySQL, PHP) — монолітна архітектура, де всі компоненти розгорнуто на єдиній платформі. Із зростанням навантаження монолітний підхід перестав справлятися з вимогами до масштабованості та надійності, що зумовило поступовий перехід до мікросервісів. Slack
Цей перехід закріпив два ключові архітектурні принципи: розподіл відповідальності — виділення окремого сервера каналів для обміну повідомленнями в реальному часі та вебзастосунку для бізнес-логіки, зберігання даних та автентифікації. Talent500
В останні роки Slack здійснив масштабну міграцію найбільш критичних сервісів до стільникової архітектури з метою збільшення надлишковості та обмеження зони впливу відмов. Основна стратегія полягає у силосуванні: кожен сервіс отримує трафік лише зсередини своєї зони доступності та надсилає трафік нагору лише до серверів у тій самій зоні. Slack
Обґрунтування вибору архітектури. Мікросервісна архітектура дозволяє масштабувати окремі компоненти незалежно один від одного залежно від поточного навантаження. Подієво-орієнтована модель забезпечує асинхронну обробку повідомлень та відокремлення виробників подій від їх споживачів. Стільникова архітектура мінімізує ризик каскадних відмов при збоях в окремих зонах хмарної інфраструктури.


3.3 Основні компоненти системи


Інтерфейс користувача. Slack надає клієнтські застосунки для усіх основних платформ: вебзастосунок у браузері, десктопний клієнт на базі фреймворку Electron (Windows, macOS, Linux) та мобільні застосунки для iOS та Android. Slack використовує поліглотний стек технологій — Go, Java, Node.js — і керовані хмарні сервіси. Клієнти підтримують синхронізацію стану між пристроями через порядкові номери подій та компактні дельти змін. Quora
Серверна частина. Серверна частина складається з декількох спеціалізованих мікросервісів:
— Gateway Server (шлюз) — точка входу, яка приймає вхідні з'єднання від клієнтів та підтримує WebSocket-сесії. У пікові години буднів Slack підтримує понад п'ять мільйонів одночасних WebSocket-сесій. ByteByteGo
— Chat Service (сервіс повідомлень) — відповідає за збереження та доставку повідомлень. Клієнт надсилає HTTP-запит до веб-API сервісу чату для публікації повідомлення; сервіс зберігає його у базі даних та делегує Dispatcher-сервісу для подальшого розповсюдження. System Design
— Dispatcher Service — виконує запит до локального сховища точок доступу для визначення центрів обробки даних підписаних клієнтів та розсилає повідомлення лише до відповідних дата-центрів через HTTP. System Design
— Presence Service (сервіс присутності) — відстежує статус онлайн/офлайн кожного користувача та поширює ці зміни у реальному часі.
— Notification Service (сервіс сповіщень) — формує та надсилає push-сповіщення на мобільні пристрої та десктоп.
— Search Service (сервіс пошуку) — забезпечує повнотекстовий пошук по повідомленнях та файлах на основі Apache Solr. Повідомлення зберігаються у MySQL, для пошуку використовується Solr, файли зберігаються на Amazon S3. stackshare
— Flannel — геодистрибутований мікросервіс, який підтримує попередньо підготовлений кеш метаданих робочого простору у пам'яті та слухає події реального часу для підтримки кешу в актуальному стані. Talent500
— Snapshot Service — прикладний мікросервіс для зниження навантаження на сервер; клієнт може звернутися до нього безпосередньо для отримання переліку користувачів у каналі або метаданих користувача; також підтримує автодоповнення у швидкому перемикачі Slack. System Design
Рівень бізнес-логіки. Бізнес-логіка розподілена між мікросервісами відповідно до принципу єдиної відповідальності. Кожен сервіс інкапсулює власний домен: управління каналами, маршрутизація повідомлень, управління підпискою, автентифікація, авторизація тощо. Взаємодія між сервісами здійснюється через REST API та черги повідомлень.
Рівень зберігання даних. Архітектура даних Slack побудована навколо Vitess — системи кластеризації баз даних для горизонтального масштабування MySQL. Кожен робочий простір (workspace) призначається до окремого шарду. Для кешування та операцій pub/sub використовується Redis. Для відкладеного опрацювання некритичних завдань, зокрема індексації повідомлень для пошуку, використовується черга завдань на базі Redis Streams або Apache Kafka. Файли користувачів зберігаються у Amazon S3. DEV CommunitySystem Design
Допоміжні сервіси. До допоміжної інфраструктури належать: система моніторингу стану сервісів, сервіс обмеження частоти запитів (rate limiting), служба управління ідемпотентними ключами для запобігання дублюванню повідомлень, а також шар інтеграцій із зовнішніми платформами через API.


3.4 Взаємодія компонентів


Slack використовує два основні протоколи для комунікації між клієнтом і сервером. Для доставки повідомлень у реальному часі використовується кастомний протокол на основі JSON-об'єктів, що передаються через WebSocket-канал (Real Time Messaging API). Для решти операцій — REST API поверх HTTPS. Quora
Клієнт надсилає HTTP-запит до веб-API сервісу чату для публікації повідомлення. Сервіс чату зберігає повідомлення у базі даних та делегує його Dispatcher-сервісу. Dispatcher виконує запит до локального сховища точок доступу для визначення дата-центрів підписаних клієнтів. Dispatcher розсилає повідомлення лише до відповідних дата-центрів через HTTP. Dispatcher у дата-центрі-адресаті визначає список шлюзових серверів, які обслуговують підписаних клієнтів. Шлюзовий сервер визначає список підписаних клієнтів та розсилає повідомлення через WebSocket. System Design
Некритичні завдання — наприклад, індексування повідомлень для пошуку або відправка email-сповіщень — передаються до черги завдань на базі Apache Kafka або Redis Streams. Воркери асинхронно опрацьовують ці завдання, не блокуючи основний потік доставки повідомлень.
Видима симптома роботи на великому масштабі — дублювання повідомлень, спричинене повторними спробами клієнта через мережеві затримки. Slack використовує ідемпотентні ключі для ідентифікації та придушення дублікатів. Клієнти синхронізують стан через порядкові номери подій та отримують лише ті зміни, яких бракує після переривання з'єднання. 3.5 Масштабованість та надійність


Вхідний трафік розподіляється між екземплярами Gateway Server за допомогою балансувальників навантаження. Усередині системи Vitess виконує роль маршрутизатора запитів до бази даних: застосунок підключається до маршрутизуючого рівня VtGate, який знає про розподіл таблиць між шардами та маршрутизує запити прозоро для застосунку — вся логіка шардування винесена з коду PHP до Vitess. InfoQ
Slack використовує горизонтальне масштабування для обробки великих обсягів даних і трафіку — шляхом додавання серверів і дата-центрів система розподіляє навантаження та залишається відповідальною і надійною. Кожен мікросервіс масштабується незалежно залежно від поточного попиту. Medium
Flannel трансформував запуск сесії з ресурсомісткої операції на операцію, керовану кешем, що підвищило надійність і масштабованість. Redis виконує роль кешу для часто запитуваних даних та забезпечує операції pub/sub для розповсюдження подій між сервісами. Talent500
Кожен шард бази даних забезпечений щонайменше двома екземплярами MySQL у різних дата-центрах, що реплікуються між собою асинхронно. При збоях з'єднання з одним хостом застосунок може повторити запит до іншого хоста без видимого впливу на клієнта, оскільки обидва вузли шарду приймають операції читання та запису. Slack
Стільникова архітектура дозволяє поступово переводити весь трафік з ураженої зони доступності протягом 5 хвилин. Реалізовано механізми throttling та шаблон «автоматичного вимикача» (circuit breaker) як засоби захисту від перевантаження запитами — вони дозволяють проактивно обмежувати або скасовувати запити до уражених шардів, підтримуючи стабільність бази даних. SlackDEV Community
Уся інфраструктура Slack розгорнута на платформі Amazon Web Services (AWS). Після мережевого збою у провайдера, що торкнувся однієї зони доступності у 2021 році та спричинив деградацію сервісу, Slack розпочав масштабну міграцію до стільникової архітектури. Завдяки розгортанню у кількох зонах доступності AWS збій в одній зоні не призводить до недоступності системи для користувачів.

4 ВИСНОВКИ


У результаті виконання практичного заняття було:
— досліджено архітектуру програмної системи Slack — хмарної платформи командної комунікації компанії Salesforce;
— визначено архітектурний стиль системи: поєднання мікросервісної, подієво-орієнтованої та стільникової архітектури на базі Amazon Web Services;
— проаналізовано основні компоненти системи: клієнтські застосунки, API Gateway, Gateway Server, Chat Service, Dispatcher, Presence, Notification, Search Service, Flannel та Snapshot Service;
— описано взаємодію компонентів: доставку повідомлень через WebSocket у реальному часі, асинхронну обробку подій через Apache Kafka та розповсюдження подій через Redis pub/sub;
— оцінено рішення масштабованості та надійності: горизонтальне масштабування мікросервісів, шардинг MySQL через Vitess, кешування сесій через Flannel, стільникову архітектуру для ізоляції збоїв та реплікацію бази даних між зонами доступності AWS;
— сформовано практичні навички архітектурного аналізу програмних систем;
— створено матеріали у репозиторії GitHub відповідно до вимог дисципліни;

  6  public class Message
  7  {
  8      public string Id          { get; set; }
  9      public string ChannelId   { get; set; }

 11      public string Text        { get; set; }

 15  public class RedisCache
 16  {
 17      private readonly Dictionary<string, List<string>> _subscriptions = new();
 18      private readonly Dictionary<string, string>       _cache         = new();
 19  
 20      public void Subscribe(string channelId, string clientId)
 21      {
 22          if (!_subscriptions.ContainsKey(channelId))
 23              _subscriptions[channelId] = new List<string>();
 24          _subscriptions[channelId].Add(clientId);
 25      }
 26  
 27      public List<string> GetSubscribers(string channelId)
 28      {
 29          return _subscriptions.TryGetValue(channelId, out var list) ? list : new List<string>();
 30      }
 31  
 32      public void Set(string key, string value) => _cache[key] = value;
 33      public string Get(string key) => _cache.TryGetValue(key, out var v) ? v : null;
 34  }
 35  
 36  public class KafkaQueue
 37  {
 38      private readonly Queue<(string Topic, Message Msg)> _queue = new();
 39  
 40      public void Publish(string topic, Message message)
 41      {
 42          _queue.Enqueue((topic, message));
 43          Console.WriteLine($"[Kafka]   Подія опублікована у топік '{topic}'");
 44      }
 45  
 46      public (string Topic, Message Msg)? Poll()
 47      {
 48          return _queue.Count > 0 ? _queue.Dequeue() : null;
 49      }
 50  }

 52  public class Database
 53  {
 54      private readonly List<Message> _messages = new();
 55  
 56      public void Save(Message message)

 58          _messages.Add(message);
 59          Console.WriteLine($"[MySQL]   Повідомлення збережено (id={message.Id}, shard за channelId={message.ChannelId})");
 60      }

 63  public class GatewayServer
 64  {
 65      public void DeliverWebSocket(string clientId, Message message)
 66      {
 67          Console.WriteLine($"[Gateway] WebSocket → клієнт '{clientId}': \"{message.Text}\"");
 68      }
 69  }
 70  
 71  public class Dispatcher
 72  {
 73      private readonly RedisCache    _cache;
 74      private readonly GatewayServer _gateway;
 75  
 76      public Dispatcher(RedisCache cache, GatewayServer gateway)
 77      {
 78          _cache   = cache;
 79          _gateway = gateway;
 80      }
 81  
 82      public void Route(Message message)
 83      {
 84          var subscribers = _cache.GetSubscribers(message.ChannelId);
 85          Console.WriteLine($"[Dispatcher] Знайдено підписників: {subscribers.Count}");
 86          foreach (var clientId in subscribers)
 87              _gateway.DeliverWebSocket(clientId, message);
 88      }
 89  }

 91  public class SearchService
 92  {
 93      public void Index(Message message)
 94      {
 95          Console.WriteLine($"[Solr]    Індексування повідомлення id={message.Id}: \"{message.Text}\"");
 96      }
 97  }
 98  
 99  public class NotificationService
100  {
101      private readonly RedisCache _cache;
102  
103      public NotificationService(RedisCache cache) => _cache = cache;
104  
105      public void Notify(Message message)
106      {
107          var status = _cache.Get($"presence:{message.ChannelId}:offline-user");
108          if (status != null)
109              Console.WriteLine($"[Notify]  Push-сповіщення → offline-user: нове повідомлення у каналі '{message.ChannelId}'");
110          else
111              Console.WriteLine($"[Notify]  Усі підписники онлайн, push не потрібен");
112      }
113  }

115  public class ChatService
116  {
117      private readonly Database    _db;
118      private readonly Dispatcher  _dispatcher;
119      private readonly KafkaQueue  _kafka;
120  
121      public ChatService(Database db, Dispatcher dispatcher, KafkaQueue kafka)
122      {
123          _db         = db;
124          _dispatcher = dispatcher;
125          _kafka      = kafka;
126      }
127  
128      public void Send(Message message)
129      {
130          Console.WriteLine($"\n[ChatService] Отримано повідомлення від '{message.SenderId}' у канал '{message.ChannelId}'");
131          _db.Save(message);
132          _dispatcher.Route(message);
133          _kafka.Publish("messages.new",    message);
134          _kafka.Publish("messages.search", message);
135      }
136  }
137  
138  public class ApiGateway
139  {
140      private readonly ChatService _chatService;
141  
142      public ApiGateway(ChatService chatService) => _chatService = chatService;
143  
144      public void HandleRequest(Message message)
145      {
146          Console.WriteLine($"[API Gateway] POST /api/chat.postMessage → ChatService");
147          _chatService.Send(message);
148      }
149  }

151  class Program
152  {
153      static async Task Main()
154      {
155          Console.WriteLine("=== Симуляція потоку даних Slack ===\n");
156  
157          var redis      = new RedisCache();
158          var kafka      = new KafkaQueue();

171  
172          Console.WriteLine("[Redis] Підписки зареєстровано: user-alice, user-bob, user-carol");
173          Console.WriteLine("[Redis] user-carol позначена як офлайн\n");
174

177              Id        = Guid.NewGuid().ToString()[..8],
178              ChannelId = "general",
179              SenderId  = "user-alice",
180              Text      = "Привіт команда! Є оновлення по проєкту.",
181              Timestamp = DateTime.UtcNow
182          };
183  
184          Console.WriteLine("--- Крок 1: Клієнт надсилає HTTP-запит ---");
185          apiGateway.HandleRequest(message);
186  
187          Console.WriteLine("\n--- Крок 6: Асинхронна обробка через Kafka ---");
188          await Task.Delay(100);
189  
190          while (true)
191          {
192              var item = kafka.Poll();
193              if (item == null) break;
194              if (item.Value.Topic == "messages.search")
195                  search.Index(item.Value.Msg);
196              else if (item.Value.Topic == "messages.new")
197                  notify.Notify(item.Value.Msg);
198          }
